165 /*
166 
167  A Database Description Packet (dd packet) has 3 parts:
168 
169  1) OSPF message header which ospf_event_send adds on and contains:
170  a) message type (OSPF_DD_MSG)
171  b) OSPF area id
172  2) DD Packet which contains:
173  a) init, more, master bits
174  b) master status (1 = master, 0 = slave)
175  c) nlsas which is # of lsa headers in this DD packet
176  3) LSA Headers, no more than OSPF_MTU will allow, and contain:
177  a) LSA age
178  b) options
179  c) LSA type
180  d) link state id
181  e) advertising router
182  f) LS sequence number
183  g) LS checksum
184  h) length
185 
186 1, 2, and 3 are each represented by membufs. 1 uses FD g_ospf_dd_fd and 2 uses FD
187 g_ospf_fd and part 3 is not sent. This is because part 2 is enough for the recvr
188 to lookup the LSA from the global LSA table and figure out what it needs. The
189 recvr should only need to compare the pointers o the sender and recvr to the table
190 to know if an LS Request should be made for the LSA. So, we don't really need to
191 send too much information.
192 
193 However, in order to do the simulation correctly, we must compute the if() block
194 on the recvr side which determines if we should create an LS request pkt. If things
195 go screwy and the if fails, even though the sender's LSA is newer, then we will
196 NOT make the request, even though we should.
197 */
